Three approaches to the material system

All replace the single dark fallback with per-group materials. They differ in structure.

A

Name-prefix map — Recommended

A single getMaterialForMesh(node) function with an ordered list of [regex, material] pairs. One traversal assigns every mesh. Adding a new material group = one line in the list. Replaces the existing two-useEffect pattern with one consolidated useEffect.

Trade-off: Single useEffect is harder to split if one material needs special lifecycle handling (e.g., animated NeoPixels). But for static materials this is cleanest.

B

Separate useEffect per group

Keep the existing pattern — one useEffect per material type. Clear separation, easy to reason about each group in isolation. Currently there are 2 effects; this would grow to ~6.

Trade-off: 6 traversals of a 300-mesh scene instead of 1. Verbose but very readable.

C

Material config object + single traversal

Same as A but the config lives in a separate modelMaterials.js file as exported constants. GlimpseModel.jsx stays lean; materials are tunable without touching the component logic.

Trade-off: Extra file for a modest benefit. Worthwhile if material values will be tweaked frequently (e.g., color tuning).
